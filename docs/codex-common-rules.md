# Codex Common Rules

이 문서는 Codex가 마음나무 프로젝트의 모든 구현 Phase에서 반드시 따라야 하는 공통 규칙이다.

## 1. 작업 시작 전 규칙

```txt
1. 먼저 현재 코드베이스의 파일 구조를 확인한다.
2. 현재 Phase와 관련된 docs 문서를 읽는다.
3. 현재 Phase의 목표와 범위를 확인한다.
4. 이미 구현된 기능과 충돌하지 않는지 확인한다.
5. 필요한 경우 TODO를 남기되, 임의로 큰 구조 변경을 하지 않는다.
```

## 2. 기술 스택 고정

다음 기술 스택을 임의로 변경하지 않는다.

```txt
Frontend: React, Vite, TypeScript
Backend: Python, FastAPI
Database: PostgreSQL
Runtime: Docker, Docker Compose
Realtime Display: SSE
```

금지 예시:

```txt
- FastAPI를 Django, Flask, NestJS 등으로 변경하지 않는다.
- PostgreSQL을 MongoDB, SQLite, Firebase 등으로 변경하지 않는다.
- React/Vite 구조를 Next.js로 변경하지 않는다.
- SSE를 WebSocket으로 임의 변경하지 않는다.
- Redis/Celery를 MVP 범위에 임의 추가하지 않는다.
```

## 3. 파일 수정 규칙

```txt
- 기존 파일을 임의로 삭제하지 않는다.
- 현재 Phase 범위 밖의 기능을 구현하지 않는다.
- 관련 없는 리팩터링을 하지 않는다.
- 파일명과 폴더 구조는 docs 기준을 따른다.
- 새 파일을 만들 때는 역할이 명확해야 한다.
- 임시 구현이 필요한 경우 TODO와 이유를 남긴다.
```

## 4. DB 구현 규칙

```txt
- DB 구조는 docs/04-data-model.md를 기준으로 한다.
- 테이블과 필드를 임의로 추가하지 않는다.
- 필요한 변경이 있으면 변경 이유를 보고한다.
- 사용자 요청에서 직접 SQL을 생성하지 않는다.
- 모든 조회는 service/repository/query function을 통해 처리한다.
- raw SQL이 필요한 경우 파라미터 바인딩을 사용한다.
- 마이그레이션 파일은 추적 가능하게 생성한다.
```

## 5. API 구현 규칙

```txt
- API 구조는 docs/05-api-spec.md를 기준으로 한다.
- 요청/응답 스키마를 명시적으로 작성한다.
- 참가자 API와 관리자 API를 구분한다.
- 관리자 API에는 인증을 적용한다.
- TV API는 원문 문장을 반환하지 않는다.
- 오류 응답 형식은 공통 error object를 사용한다.
- idempotent가 필요한 API는 중복 호출에 안전해야 한다.
```

## 6. 세션 구현 규칙

```txt
- 참가자는 회원가입하지 않는다.
- 원문 resume token 또는 session key를 DB에 저장하지 않는다.
- 세션 복구는 hash 기반으로 처리한다.
- 세션 상태 전이는 백엔드 서비스 로직에서 처리한다.
- 프론트엔드 상태만 믿고 완료 처리하지 않는다.
- 완료 코드는 세션당 1개만 발급한다.
```

## 7. 점수화와 위험 플래그 규칙

```txt
- 척도 점수화는 서버가 최종 계산한다.
- 프론트엔드는 점수화 결과를 최종 판단값으로 만들지 않는다.
- 위험 플래그는 서버 규칙으로 계산한다.
- 점수화 기준은 docs/09-scoring-risk-policy.md와 docs/data/questions_fire_expo_2026.json을 따른다.
- rule_version은 v2-2026-05-13-scale-cutoffs을 사용한다.
- K-PHQ-9 item9, 자유입력 위기 표현, K-PCL-5, K-MIES 기준을 명시적으로 처리한다.
- K-SCS는 지정 문항을 역채점한 뒤 평균값으로 수준을 판정한다.
- LLM이 위험도, 자살위험, 공개 여부를 판단하도록 구현하지 않는다.
```

척도 절단점 요약:

```txt
K-PHQ-9:
- 0~4 특이소견 없음
- 5~9 가벼운 우울증상
- 10~19 중간 정도 우울증 의심
- 20~27 심한 우울증 의심 점수 구간
- 16점 이상 불안정도 높음 신호
- 22번 item9 >= 1이면 위기관리/도움 안내 필요

K-PCL-5:
- 0~30 정상범위
- 31~33 임계치
- 34~80 고위험군

K-MIES:
- 9~18 낮음
- 19~36 중간
- 37~54 높음

K-SCS:
- local item 2,3,5,6,8,14,15,19,20,23,25 역채점
- global question no 53,54,56,57,59,65,66,70,71,74,76 역채점
- 평균 1.0~2.4 낮은 수준
- 평균 2.5~3.5 보통
- 평균 3.6~5.0 높음
```

## 8. LLM 구현 규칙

LLM은 보조 도구다.

```txt
LLM이 할 수 있는 것:
- 마음신호 요약 문장 보정
- 마음카드 키워드 후보 추출
- 응원 문장 키워드 후보 추출
- 문장 톤 정리

LLM이 하면 안 되는 것:
- 진단
- 위험도 최종 판정
- 자살위험 최종 판단
- 공개 여부 결정
- 관리자 승인 대체
- DB 구조 결정
- API 구조 결정
- 세션 상태 결정
```

구현 기준:

```txt
- LLM 처리는 참가자 흐름을 막지 않는다.
- LLM 실패 시 fallback 키워드 추출을 사용한다.
- LLM provider는 환경변수로 제어한다.
- 실제 API key를 코드에 넣지 않는다.
- LLM timeout과 retry 정책을 둔다.
- LLM 결과와 fallback 여부를 keyword_jobs에 기록한다.
```

## 9. 프론트엔드 구현 규칙

```txt
- 참가자, TV, 관리자 라우트를 분리한다.
- 참가자 진입 시 1~77번 문항 전체를 선로딩한다.
- 문항 진행은 로컬 상태 기준으로 부드럽게 처리한다.
- 제출 실패 시 사용자의 입력값을 유지한다.
- 재시도 버튼을 제공한다.
- 완료 후 민감한 로컬 임시 데이터를 정리한다.
- TV 화면은 SSE 끊김 시 마지막 데이터를 유지한다.
- 관리자 화면에서 원문은 필요한 범위에서만 표시한다.
```

## 10. 보안·개인정보 규칙

```txt
- 이름, 전화번호, 소속 등 직접 식별정보는 기본 수집하지 않는다.
- 참가자에게 실명, 소속, 연락처, 구체적 장소, 날짜, 사건명을 입력하지 말라고 안내한다.
- 개인정보 또는 식별 가능한 사건 정보가 감지되면 공개 제한한다.
- TV에는 원문, 점수, 위험 플래그, 세션 정보를 표시하지 않는다.
- 관리자 주요 행위는 감사 로그로 남긴다.
- 실제 운영 secret, API key, DB password를 코드에 넣지 않는다.
- .env.example에는 placeholder만 둔다.
```

## 11. Docker 구현 규칙

```txt
- docker-compose.yml은 postgres, api, web 컨테이너를 기본으로 한다.
- 개발용 .env.example을 제공한다.
- docker compose up으로 기본 실행이 가능해야 한다.
- API health check를 제공한다.
- DB migration 실행 방법을 README 또는 Phase 보고에 포함한다.
```

## 12. 테스트 규칙

각 Phase는 가능한 범위에서 테스트 방법을 제공해야 한다.

```txt
- 백엔드 단위 테스트
- API smoke test
- 프론트엔드 기본 렌더링 확인
- Docker 실행 확인
- 세션 재접속 확인
- 제출 재시도 확인
- LLM disabled/mock 모드 확인
- fallback 키워드 확인
- SSE 자동 재연결 확인
- 완료 코드 중복 지급 방지 확인
```

## 13. 작업 후 보고 형식

Codex는 작업 후 반드시 아래 형식으로 보고한다.

```md
## 작업 요약
- ...

## 변경 파일
- path/to/file: 변경 이유

## 실행 방법
- ...

## 테스트 결과
- ...

## 남은 작업 / TODO
- ...

## 주의 사항
- ...
```

## 14. 절대 금지 사항

```txt
- 기존 파일을 임의 삭제하지 않는다.
- 현재 Phase 범위를 넘어서는 기능을 구현하지 않는다.
- 기술 스택을 임의로 바꾸지 않는다.
- 실제 API key를 코드에 넣지 않는다.
- 사용자 입력으로 직접 SQL을 생성하지 않는다.
- LLM에 위험 판단이나 공개 여부 판단을 맡기지 않는다.
- TV API에서 원문 문장을 반환하지 않는다.
- 관리자 인증 없이 원문 조회 API를 만들지 않는다.
- 임시 구현을 완료 구현처럼 보고하지 않는다.
```
