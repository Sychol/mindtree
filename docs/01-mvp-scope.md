# 01. MVP Scope

## 1. MVP 목표

마음나무 MVP의 목표는 기능을 많이 넣는 것이 아니라, 박람회 현장에서 끊기지 않고 운영되는 이벤트 서비스를 만드는 것이다.

MVP는 다음 세 문장을 기준으로 한다.

```txt
1. 참가자는 네트워크와 LLM 상태에 상관없이 최대한 끊김 없이 완료까지 진행해야 한다.
2. LLM은 사용자 흐름을 막지 않는 비동기 보조 모듈로 사용하고, 실패 시 규칙 기반 fallback을 사용한다.
3. TV 마음나무는 실시간성을 지향하되, SSE 재연결과 비동기 키워드 반영을 전제로 안정적으로 운영한다.
```

```txt
- 문항 응답 로컬 진행 상태 유지
- 네트워크 실패 시 제출 재시도
- 척도 응답 저장
- 서버 기준 척도 점수화
- 규칙 기반 위험 플래그 계산
- 템플릿 기반 마음신호 요약
- 선택적 LLM 요약 보정
- 마음카드 작성
- 공개 가능한 타인 마음카드 선택
- 응원·공감·작은 대처법 작성
- 완료 코드 발급
```

## 3. MVP에서 제외할 기능

아래 기능은 MVP에서 구현하지 않는다.

```txt
- RAG 후보 승인/반려
- RAG 데이터 export
- pgvector 기반 유사 사례 검색
- 기관형 멀티테넌트 고도화
- 상담 라우팅
- 장기 사용자 히스토리
- 회원 기반 리본톡 본 서비스 연동
- Redis/Celery 기반 작업 큐
- 복잡한 관리자 통계 대시보드
- 자동 리포트 생성
- 고도화된 자연어 상담 챗봇
- 실시간 관리자 알림 시스템
- SMS/카카오톡/이메일 발송
```

단, Redis/Celery를 제외하더라도 PostgreSQL 기반 `keyword_jobs` 테이블은 MVP에 포함한다. 이 테이블은 LLM 지연, 실패, 재처리 상태를 현장에서 확인하기 위한 최소 운영 장치다.

## 4. 현장 운영 기본 조건

## 4.1 네트워크 조건

박람회장에서는 모바일 네트워크가 불안정할 수 있다. MVP는 다음 상황을 정상 운영 조건으로 간주한다.

```txt
- LTE/5G 연결 지연
- 일시적인 네트워크 끊김
- 사용자의 새로고침
- 사용자의 중간 이탈 후 재접속
- TV SSE 연결 끊김
```

이에 대한 구현 기준은 다음이다.

```txt
- 문항은 초기 진입 시 전체 선로딩한다.
- 문항 이동은 로컬 상태 기준으로 처리한다.
- 서버 제출 실패 시 재시도 가능해야 한다.
- 같은 응답을 여러 번 제출해도 중복 저장되지 않아야 한다.
- TV는 연결 끊김 시 마지막 데이터를 유지해야 한다.
- TV는 자동 재연결해야 한다.
```

## 4.2 LLM 조건

MVP는 다음 상황을 정상 운영 조건으로 간주한다.

```txt
- LLM API 응답 지연
- LLM API rate limit
- LLM API 실패
- LLM provider 비활성화
- 운영 중 mock 또는 disabled 모드 사용
```

이에 대한 구현 기준은 다음이다.

```txt
- 참가자 흐름은 LLM을 기다리지 않는다.
- 마음신호 요약은 템플릿 기반으로 즉시 제공 가능해야 한다.
- 키워드 추출은 서버 내부 비동기 job으로 처리한다.
- LLM 실패 시 fallback 키워드를 사용한다.
- 실패 job은 관리자 화면에서 확인하고 재시도할 수 있어야 한다.
```

## 5. 문항 범위와 seed 파일

MVP는 CSV 기준 1~77번 문항을 기준으로 한다.

```txt
1~13번   profile / 인구통계·직무·사건 기준 문항
14~22번  phq9 / K-PHQ-9
23~42번  pcl5 / K-PCL-5
43~51번  kmies / K-MIES
52~77번  kscs / K-SCS
```

문항은 하드코딩하지 않고 아래 JSON seed에서 로딩한다.

```txt
docs/data/questions_fire_expo_2026.json
docs/data/scoring_rules_v1.json
```

JSON에는 문항, 선택지, scoreMap, 척도 목적, 절단점, K-SCS 역채점 문항이 포함된다. 프론트엔드는 API로 받은 문항 설정을 기반으로 동적으로 렌더링하고, 최종 점수화는 서버가 수행한다.

## 6. 척도 절단점과 위험 플래그 범위

MVP의 점수화와 위험 플래그는 서버 규칙으로 계산한다. 기준 rule version은 다음이다.

```txt
v2-2026-05-13-scale-cutoffs
```

### 6.1 K-PHQ-9

```txt
목적:
- 성인의 우울 정도를 측정하는 자가보고 설문지

문항:
- 14~22번

점수 범위:
- 0~27점

절단점:
- 0~4점: 특이소견 없음
- 5~9점: 가벼운 우울증상
- 10~19점: 중간 정도 우울증 의심
- 20~27점: 심한 우울증 의심 점수 구간

추가 규칙:
- 16점 이상: 불안정도 높음 신호를 details에 기록
- 22번 문항 item9가 1점 이상: phq9_item9_positive, help_notice_required, public_restriction true
- 20점 이상이면서 item9 1점 이상: phq9_severe_with_item9를 details에 기록
```

### 6.2 K-PCL-5

```txt
목적:
- 지난 한 달 동안 겪은 고통스러운 사건으로 인한 PTSD 증상의 심각도 측정

문항:
- 23~42번

점수 범위:
- 0~80점

절단점:
- 0~30점: 정상범위
- 31~33점: 임계치
- 34~80점: 고위험군

위험 플래그:
- 34점 이상이면 trauma_high_signal true
- 31~33점은 details.pcl5_threshold_signal true
```

### 6.3 K-MIES

```txt
목적:
- 재난 현장에서 도덕적 신념에 어긋나는 상황을 겪은 후 느끼는 심리적 고통 측정

문항:
- 43~51번

점수 범위:
- 9~54점

절단점:
- 9~18점: 낮음
- 19~36점: 중간
- 37~54점: 높음

위험 플래그:
- 37점 이상이면 moral_injury_high_signal true
```

### 6.4 K-SCS

```txt
목적:
- 고통스러운 상황에서 자신을 얼마나 친절하고 객관적으로 대하는지를 측정

문항:
- 52~77번

점수화:
- 2, 3, 5, 6, 8, 14, 15, 19, 20, 23, 25번 local item 역채점
- CSV/global question no 기준 53, 54, 56, 57, 59, 65, 66, 70, 71, 74, 76번 역채점
- 역채점 공식: 6 - answer_value
- 역채점 후 평균값 사용

절단점:
- 1.0~2.4점: 낮은 수준
- 2.5~3.5점: 보통
- 3.6~5.0점: 높음
```

주의:

```txt
- LLM은 위험 플래그를 최종 판단하지 않는다.
- 위험 플래그가 있다는 이유만으로 참가자 완료나 상품 지급을 자동 차단하지 않는다.
- TV에는 점수, 구간, 위험 플래그, 원문을 표시하지 않는다.
```

## 7. 공개 정책 범위

```txt
TV 표시 가능:
- 정제된 익명 키워드
- 참여자 수
- TOP 키워드
- 이벤트 안내 문구

TV 표시 금지:
- 참가자 개인 결과
- 척도 점수
- 위험 플래그

- 응원 문장 원문

- 관리자 검수 상태
- 위기 표현
```

## 8. MVP 완료 기준

MVP는 다음 조건을 만족해야 완료로 본다.

```txt
1. docker compose up으로 postgres, api, web이 실행된다.
2. 참가자는 QR/URL 진입부터 완료 코드까지 진행할 수 있다.
3. 문항 1~77번이 API 기반으로 로딩되고 응답 저장이 가능하다.
4. 서버가 점수화와 위험 플래그를 계산한다.
5. 마음신호 요약은 LLM 없이도 생성된다.
6. 마음카드와 응원 문장이 저장되고 검수 상태가 부여된다.
7. keyword job이 생성되고 LLM 또는 fallback으로 키워드가 저장된다.
8. TV 화면은 SSE로 키워드 집계를 수신한다.
9. SSE 연결이 끊겨도 마지막 데이터가 유지되고 재연결된다.
10. 관리자는 검수, keyword job 상태 확인, 완료 코드 지급 처리를 할 수 있다.
11. TV에는 개인 결과와 원문이 표시되지 않는다.
12. 관리자 주요 행위가 감사 로그로 남는다.
```
