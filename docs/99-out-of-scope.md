# 99. Out of Scope

## 1. 목적

이 문서는 마음나무 MVP에서 의도적으로 제외하는 기능을 명확히 정의한다.

MVP의 목표는 박람회 현장에서 끊기지 않고 운영되는 익명 참여형 마음 점검·응원 이벤트다. 따라서 구현 범위를 벗어나는 기능을 Codex가 임의로 추가하지 않도록 이 문서를 기준으로 한다.

## 2. MVP 핵심 범위 재확인

MVP에 집중해야 하는 범위는 다음이다.

```txt
- QR/URL 접속
- 익명 세션 생성과 복구
- 필수 동의
- 1~77번 문항 선로딩과 응답
- 서버 점수화와 위험 플래그
- 템플릿 기반 마음신호 요약
- 마음카드 작성
- 타인 카드 선택
- 응원 문장 작성
- 비동기 키워드 추출
- fallback 키워드
- TV 마음나무 SSE display
- 관리자 검수
- 완료 코드와 상품 지급
- Docker Compose 실행
```

이 범위를 벗어나는 기능은 후속 Phase로 분리한다.

## 3. 제외 기능 목록

## 3.1 RAG 후보 승인/반려

MVP에서 제외한다.

제외 이유:

```txt
- 현장 이벤트 운영에 필수 기능이 아니다.
- 동의 문구와 데이터 활용 범위를 복잡하게 만든다.
- 관리자 검수 범위가 커진다.
- pgvector 또는 별도 검색 구조가 필요해질 수 있다.
```

후속 조건:

```txt
- 비식별 처리 정책 확정
- 별도 동의 문구 확정
- 후보 데이터 export/삭제 정책 확정
- 관리자 승인 workflow 확정
```

## 3.2 RAG 데이터 export

MVP에서 제외한다.

제외 이유:

```txt
- 운영 후 분석 기능이다.
- 현장 이벤트 완료에는 필요하지 않다.
- 원문/비식별 데이터 관리 리스크가 있다.
```

## 3.3 pgvector 기반 유사 사례 검색

MVP에서 제외한다.

제외 이유:

```txt
- MVP는 실시간 체험 이벤트다.
- 벡터 검색은 데이터 품질과 보존 정책이 확정된 후 도입해야 한다.
- PostgreSQL 기본 테이블과 keyword job만으로 MVP 운영이 가능하다.
```

## 3.4 Redis/Celery 기반 작업 큐

MVP에서 제외한다.

제외 이유:

```txt
- 운영 구성 요소가 늘어난다.
- 박람회 MVP에서는 PostgreSQL keyword_jobs로 충분히 시작할 수 있다.
- 장애 지점이 증가한다.
```

단, PostgreSQL 기반 `keyword_jobs`는 MVP에 포함한다.

후속 조건:

```txt
- 동시 처리량 증가
- job 지연 증가
- API 프로세스와 worker 분리 필요
- 운영 모니터링 필요
```

## 3.5 WebSocket 기반 실시간 통신

MVP에서 제외한다.

제외 이유:

```txt
- TV는 서버에서 클라이언트로 단방향 업데이트만 필요하다.
- SSE가 MVP 요구에 더 단순하다.
- WebSocket은 연결 관리와 배포 구성이 복잡하다.
```

## 3.6 기관형 멀티테넌트

MVP에서 제외한다.

제외 이유:

```txt
- 현재 목표는 단일 이벤트 운영이다.
- 기관별 권한, 데이터 분리, 리포트 정책이 필요해진다.
- 관리자 구조가 복잡해진다.
```

후속 조건:

```txt
- organization 테이블 도입
- event ownership 정책
- 관리자 role/permission 세분화
- 기관별 데이터 보존 정책
```

## 3.7 상담 라우팅

MVP에서 제외한다.

제외 이유:

```txt
- 마음나무는 진단/상담/치료 서비스가 아니다.
- 상담 연결은 별도 운영 책임과 안전 프로토콜이 필요하다.
- 현장 이벤트 완료 흐름을 복잡하게 만든다.
```

MVP에서는 도움 안내 문구만 제공한다.

## 3.8 장기 사용자 히스토리

MVP에서 제외한다.

제외 이유:

```txt
- 참가자는 회원가입하지 않는다.
- 장기 추적은 개인정보와 동의 범위가 달라진다.
- 이벤트성 익명 세션 원칙과 맞지 않는다.
```

## 3.9 회원 기반 리본톡 본 서비스 연동

MVP에서 제외한다.

제외 이유:

```txt
- 외부 또는 본서비스 계정 연동은 인증/동의/데이터 이전 이슈가 있다.
- 박람회 현장에서는 회원가입 없는 빠른 참여가 우선이다.
```

## 3.10 복잡한 관리자 통계 대시보드

MVP에서 제외한다.

제외 이유:

```txt
- 현장 운영에 필요한 지표는 참여자 수, 완료 수, 검수 수, job 실패 수, 지급 수 정도다.
- 복잡한 통계는 구현 시간을 늘리고 개인정보 관리 범위를 넓힌다.
```

## 3.11 자동 리포트 생성

MVP에서 제외한다.

제외 이유:

```txt
- 이벤트 후 분석 기능이다.
- 데이터 해석과 표현 정책이 별도로 필요하다.
- 진단처럼 오해될 수 있는 리포트 문구 검토가 필요하다.
```

## 3.12 실시간 관리자 알림

MVP에서 제외한다.

제외 이유:

```txt
- 알림 채널 구축이 필요하다.
- 관리자 대시보드에서 수동 확인으로 충분히 시작할 수 있다.
```

## 3.13 SMS/카카오톡/이메일 발송

MVP에서 제외한다.

제외 이유:

```txt
- 전화번호나 이메일 수집이 필요해질 수 있다.
- 개인정보 최소 수집 원칙과 충돌한다.
- 외부 발송 서비스 연동이 필요하다.
```

## 3.14 결제/쿠폰 시스템

MVP에서 제외한다.

제외 이유:

```txt
- 상품 지급은 완료 코드 수동 확인으로 충분하다.
- 외부 쿠폰/결제 연동은 운영 리스크가 크다.
```

## 3.15 고도화된 자연어 상담 챗봇

MVP에서 제외한다.

제외 이유:

```txt
- 마음나무는 상담 챗봇이 아니다.
- 안전 정책과 응답 검증이 크게 복잡해진다.
- LLM이 위험 판단이나 조언을 하게 될 가능성이 있다.
```

## 4. Codex 구현 금지 기준

Codex는 다음 기능을 현재 Phase 범위에 임의 추가하지 않는다.

```txt
- RAG 후보 테이블 추가
- vector extension 또는 embedding 저장소 추가
- Redis/Celery 추가
- WebSocket 서버 추가
- 기관/조직 멀티테넌트 구조 추가
- 상담사 배정 기능 추가
- 회원가입/로그인 기능 추가
- SMS/이메일 발송 기능 추가
- 자동 보고서 생성 기능 추가
- 복잡한 통계 chart 대시보드 추가
```

필요성이 보이면 TODO와 이유를 남기고 구현하지 않는다.

## 5. 후속 Phase 후보

MVP 이후 확장 Phase는 별도 문서에서 다룬다.

```txt
PHASE X1. RAG 후보 데이터 비식별화와 승인 workflow
PHASE X2. pgvector 기반 유사 키워드/문장 검색
PHASE X3. 기관형 멀티테넌트 구조
PHASE X4. 고급 관리자 통계와 이벤트 리포트
PHASE X5. 상담/도움 연계 프로토콜
PHASE X6. Redis/Celery 기반 worker 분리
PHASE X7. 운영 배포 보안 고도화
```

## 6. 충돌 해결 원칙

새 요구사항이 MVP 제외 범위와 충돌하면 다음 순서로 판단한다.

```txt
1. 현장 이벤트 완료에 반드시 필요한가?
2. 개인정보 또는 안전 정책을 복잡하게 만들지 않는가?
3. 기존 DB/API/세션 구조를 크게 바꾸지 않는가?
4. Docker Compose 기본 실행을 복잡하게 만들지 않는가?
5. LLM에 판단 책임을 넘기지 않는가?
```

위 조건을 만족하지 못하면 MVP에 넣지 않는다.

## 7. 정리

MVP에서 중요한 것은 기능 수가 아니라 현장 운영 안정성이다.

```txt
포함할 것:
- 끊기지 않는 참가자 흐름
- 안전한 공개 제한
- 비동기 키워드 처리
- TV SSE display
- 관리자 검수와 상품 지급

제외할 것:
- RAG
- pgvector
- 기관형 서비스
- 상담 라우팅
- 장기 히스토리
- 복잡한 통계
- 외부 발송/연동
```
